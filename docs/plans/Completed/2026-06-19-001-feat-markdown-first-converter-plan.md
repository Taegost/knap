# Markdown-First Frontmatter Converter

## Summary

Create a reusable converter script that transforms markdown file frontmatter from manual string-based formatting to `yaml.dump()` output. The script processes all `.md` files in the repo, preserves data integrity through round-trip consistency, and includes comprehensive inline documentation for use in both init workflows and one-time repo fixes. This is a prerequisite for the Typed Links plan.

## Problem Frame

The current ingestion script (`ingest.py`) serializes frontmatter fields using manual string formatting (`f"  - {item}"`). This works for simple string lists but produces Python repr for dicts, force-quotes all strings, and doesn't follow markdown-first conventions. The Typed Links plan needs `yaml.dump()` for all frontmatter to properly serialize list-of-dict structures like `links`, but switching serialization without converting existing pages would create formatting inconsistencies between old and new pages.

The repo contains files with frontmatter across 6 categories (wiki pages, raw files, plans, brainstorms, skills, docs). Wiki pages are the primary conversion target — they're generated by `ingest.py` with manual formatting. Other files already use clean YAML but benefit from consistent formatting.

## Requirements

- R1. The converter script processes all `.md` files in the repo, skipping files without frontmatter.
- R2. Frontmatter is re-serialized using `yaml.dump()` with `default_flow_style=False` and `sort_keys=False` to preserve field order.
- R3. Round-trip consistency: `read → yaml.safe_load → yaml.dump → write → read → yaml.safe_load` produces identical data structures.
- R4. The script includes a `--dry-run` flag to preview changes without writing, matching the existing `ingest.py` convention.
- R5. Per-file error handling: the script continues processing remaining files when one file fails, reporting successes and failures.
- R6. The script includes comprehensive inline documentation explaining purpose, usage, edge cases, and system integration.
- R7. Run `validate.py` before conversion to get a baseline (some files may already fail). Run again after conversion — files must be in the same or better state than before; no previously-passing file may fail.

## Key Technical Decisions

**KTD-1: Standalone script, not integrated into ingest.py.** The converter is a separate script in `.knap/scripts/` because it serves a different purpose (batch conversion of existing files) than ingestion (creating new wiki pages from raw files). It can be run independently during init or as a one-time fix. Rationale: separation of concerns; the converter is a utility, not part of the ingestion pipeline.

**KTD-2: Process all .md files, skip those without frontmatter.** The script scans the entire repo for `.md` files. Files without frontmatter (no opening `---`) are skipped silently. Files with frontmatter are parsed, re-serialized, and written back. Rationale: the user explicitly requested all .md files; skipping non-frontmatter files is safe and avoids accidental modification.

**KTD-3: Use `yaml.dump()` with `default_flow_style=False, sort_keys=False`.** This matches the existing pattern in `schema.py` line 22. `default_flow_style=False` produces block-style YAML (one key per line). `sort_keys=False` preserves the original field order. Rationale: consistency with existing codebase conventions; block style is more readable than flow style.

**KTD-4: Preserve data types through round-trip.** `yaml.safe_load()` parses dates as `datetime.date` objects, booleans as `bool`, strings as `str`, etc. `yaml.dump()` serializes these back to their YAML representations. The converter must not alter data types — only formatting. Rationale: data integrity is the core requirement; formatting changes must not introduce semantic differences.

**KTD-5: Per-file error handling with continue-on-failure.** If a file fails to parse or write, the script logs the error and continues with the next file. This is more useful for batch operations than halt-on-first-error. Rationale: a single malformed file shouldn't block conversion of the entire repo.

**KTD-6: Comprehensive inline documentation and code comments.** The script includes: a module docstring with purpose, usage examples, and edge cases; inline comments explaining key decisions and the relationship to the Typed Links plan; and code-level comments on each logical block so someone unfamiliar with the script can understand what each piece does without reading the whole file. Rationale: the user explicitly requested "very good inline documentation" for a reusable tool that will be used by people who may not be familiar with Python or the knap system.

## Implementation Units

### U1. Create the converter script

- **Goal:** A reusable script that converts markdown frontmatter to `yaml.dump()` format across all `.md` files in the repo.
- **Requirements:** R1, R2, R3, R4, R5, R6, R7
- **Dependencies:** None
- **Files:**
  - `.knap/scripts/convert_frontmatter.py` — new: frontmatter format converter
- **Approach:** Create a standalone script with the following components:

  **CLI interface:** Use `argparse` with `--dry-run` flag (default: off) and optional `--path` argument to limit conversion to a specific directory. Default behavior: scan entire repo.

  **File discovery:** Walk the repo directory tree, find all `.md` files. Skip these non-content directories: `.venv/`, `.git/`, `__pycache__/`, `node_modules/`, `.claude/`. Skip files without frontmatter (no opening `---` at line 1).

  **Frontmatter parsing:** Use the same pattern as `ingest.py` lines 140-151: find `---` at start, find next `---`, extract YAML between them with `yaml.safe_load()`. Handle `ValueError` for malformed frontmatter and `yaml.YAMLError` for invalid YAML.

  **Serialization:** Re-serialize the parsed frontmatter dict using `yaml.dump(data, default_flow_style=False, sort_keys=False)`. Reconstruct the file as: `---\n` + yaml_dump output + `---\n` + original body content.

  **Round-trip verification:** Write the converted content to a temporary file first. Smoke test: re-read the temp file, verify `yaml.safe_load()` produces the same data structure as the original parse, and verify the body content (everything after the closing `---`) is byte-identical to the original. Only if both checks pass, replace the original file with the temp file. If either check fails, log an error and delete the temp file. Preserve original line endings throughout — read with `newline=''` and write with the same line ending style detected in the original file.

  **Validate.py integration:** Run `validate.py` before conversion to capture a baseline (some files may already fail). After all conversions complete, run `validate.py` again. Compare results — no previously-passing file may fail. Report any regressions.

  **Error handling:** Wrap each file's processing in a try/except. On failure, log the error to stderr and continue. Track success/failure counts.

  **Reporting:** Print per-file status (converted, skipped, failed) and a summary at the end.

  **Inline documentation:** Include a module docstring with purpose, usage examples, and edge cases. Add inline comments explaining key decisions and the relationship to the Typed Links plan.

- **Test scenarios:**
  - Happy path: a wiki page with manual string formatting is converted to `yaml.dump()` format
  - Happy path: a plan file with clean YAML is re-serialized without data loss
  - Happy path: a file without frontmatter is skipped
  - Edge case: a file with `source_url: "n/a"` preserves the value
  - Edge case: a file with bare YAML dates (`date_farmed: 2026-06-17`) preserves the date
  - Edge case: a file with markdown links in strings (`source: "[file.md](../raw/...)"`) preserves the link
  - Edge case: a file with empty lists (`tags: []`) preserves the empty list
  - Error path: a file with malformed frontmatter logs an error and continues
  - Dry-run: `--dry-run` flag shows what would change without writing
  - Round-trip: converted file re-reads to identical data structure
  - Validation: `validate.py` baseline captured before conversion; after conversion, no previously-passing file fails
- **Verification:** Run `validate.py` before conversion to get a baseline (some files may already fail). Run the script on the repo, confirm all files with frontmatter are converted, confirm round-trip consistency, confirm `--dry-run` shows changes without writing. Run `validate.py` after conversion — verify no previously-passing file now fails.

## Scope Boundaries

**In scope:**
- Converter script with CLI interface
- File discovery and frontmatter detection
- `yaml.dump()` serialization with round-trip consistency
- Dry-run mode and per-file error handling
- Comprehensive inline documentation
- Unit tests for the converter script
- `validate.py` integration before and after conversion

**Deferred for later:**
- Updating `ingest.py` to use `yaml.dump()` for new file creation — that's the Typed Links plan's responsibility
- Converting files in directories outside the repo (e.g., external wikis)

**Deliberately excluded:**
- Modifying `ingest.py` — the converter is a prerequisite, not a replacement
- Adding new fields or features — this is a formatting change only
- Converting files without frontmatter — those are intentionally frontmatter-free
